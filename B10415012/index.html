Page tables(代碼解釋）

Code: creating an address space
main調用kvmalloc（1757），創建並切換到一個擁有內核運行所需的KERNBASE以上映射的頁表。這裡的大多數工作都是由setupkvm（1737）完成的。首先，它會分配一頁內存來放置頁目錄，然後調用mappages來建立內核需要的映射，這些映射可以在kmap（1728）數組中找到。這裡的映射包括內核的指令和數據，PHYSTOP以下的物理內存，以及I/O設備所佔的內存。setupkvm不會建立任何用戶內存的映射，這些映射稍後會建立。
mappages（1679）做的工作是在頁表中建立一段虛擬內存到一段物理內存的映射。它是在頁的級別，即一頁一頁地建立映射的。對於每一個待映射虛擬地址，mappages調用walkpgdir來找到該地址對應的PTE地址。然後初始化該PTE以保存對應物理頁號、許可級別（PTE_W和/或PTE_U）以及PTE_P位來標記該PTE是否是有效的（1691）。
walkpgdir（1654）模仿x86的分頁硬件為一個虛擬地址尋找PTE的過程。walkpgdir通過虛擬地址的前10位來找到在頁目錄中的對應條目（1659），如果該條目不存在，說明要找的頁表頁尚未分配；如果alloc參數被設置了，walkpgdir會分配頁表頁並將其物理地址放到頁目錄中。最後用虛擬地址的接下來10位來找到其在頁表中的PTE地址（1672）。

Code: Physical memory allocator
分配器中的數據結構是一個由可分配物理內存頁構成的空閒鍊錶。這個空閒頁的鍊錶的元素是結構體struct run（2764）。那麼分配器從哪裡獲得內存來存放這些數據結構呢？實際上，分配器將每個空閒頁的run結構體保存在該空閒頁本身中，因為空閒頁中沒有其他數據。分配器還用一個spin lock（2764-2766）來保護空閒鍊錶。鍊錶和這個鎖都封裝在一個結構體中，這樣邏輯就比較明晰：鎖保護了該結構體中的域。不過現在讓我們先忽略這個鎖，以及對acquire和release的調用
main函數調用了kinit1和kinit2兩個函數對分配器進行初始化（2780）。這樣做是由於main中的大部分代碼都不能使用鎖以及4MB以上的內存。kinit1在前4MB進行了不需要鎖的內存分配。而kinit2允許了鎖的使用，並使得更多的內存可用於分配。原本應該由main決定有多少物理內存可用於分配，但在x86上很難實現。所以它假設機器中有240MB（PHYSTOP）物理內存，並將內核末尾和PHYSTOP之間的內存都作為一個初始的空閒內存池。kinit1和kinit2調用freerange將內存加入空閒鍊錶中，freerange則是通過對每一頁調用kfree實現該功能。一個PTE只能指向一個4096字節對齊的物理地址（即是4096的倍數），因此freerange用PGROUNDUP來保證分配器只會釋放對齊的物理地址。分配器原本一開始沒有內存可用，正是對kfree的調用將可用內存交給了分配器來管理。
分配器用映射到高內存區域的虛擬地址找到對應的物理頁，而非物理地址。所以kinit會使用p2v(PHYSTOP)來將PHYSTOP（一個物理地址）翻譯為虛擬地址。分配器有時將地址看作是整型，這是為了對其進行運算（譬如在kinit中遍歷所有頁）；而有時將地址看作讀寫內存用的指針（譬如操作每個頁中的run結構體）；對地址的雙重使用導致分配器代碼中充滿了類型轉換。另外一個原因是，釋放和分配內存隱性地改變了內存的類型。

函數kfree（2815）首先將被釋放內存的每一字節設為1。這使得訪問已被釋放內存的代碼所讀到的不是原有數據，而是垃圾數據；這樣做能讓這種錯誤的代碼儘早崩潰。接下來kfree把v轉換為一個指向結構體struct run的指針，在r->next中保存原有空閒鍊錶的表頭，然後將當前的空閒鍊錶設置為r。kalloc移除並返回空閒鍊錶的表頭。

Code: sbrk
sbrk是Linux上的系統API，Windows上使用HeapAlloc和HeapFree來管理堆區。

sbrk調用時，它分配的新的物理記憶體頁，更新進程的頁表，並設置程序中斷點。

理論上，程序應該直接調用sbrk（而不是通過malloc），並且自己管理堆區。但是malloc易於使用，並且對於大多數記憶體使用模式，它運行速度快並且高效利用記憶體。

為了實現記憶體管理API，多數Linux系統都使用ptmalloc，它基於dlmalloc，由Doug Lea編寫。
Code: exec
exec是創建地址空間中用戶部分的系統調用。它根據文件系統中保存的某個文件來初始化用戶部分。exec（5910）通過namei（5920）打開二進製文件，這一點將在第6章進行解釋。然後，它讀取ELF頭。xv6應用程序以通行的ELF格式來描述，該格式在elf.h中定義。一個ELF二進製文件包括了一個ELF頭，即結構體struct elfhdr（0955），然後是連續幾個程序段的頭，即結構體struct proghdr（0974）。每個proghdr都描述了需要載入到內存中的程序段。xv6中的程序只有一個程序段的頭，但其他操作系統中可能有多個。

exec第一步是檢查文件是否包含ELF二進制代碼。一個ELF二進製文件是以4個“魔法數字”開頭的，即0x7F，“E”，“L”，“F”，或者寫為宏ELF_MAGIC（0952）。如果ELF頭中包含正確的魔法數字，exec就會認為該二進製文件的結構是正確的。

exec通過setupkvm（5931）分配了一個沒有用戶部分映射的頁表，再通過allocuvm（5943）為每個ELF段分配內存，然後通過loaduvm（5945）把段的內容載入內存中。allocuvm會檢查請求分配的虛擬地址是否是在KERNBASE之下。loaduvm（1818）通過walkpgdir來找到寫入ELF段的內存的物理地址；通過readi來將段的內容從文件中讀出。

exec創建的第一個用戶程序/init程序段的頭是這樣的：

#objdump -p _init

_init: file format elf32-i386

Program Header:
LOAD off 0x00000054 vaddr 0x00000000 paddr 0x00000000 align 2**2
filesz 0x000008c0 memsz 0x000008cc flags
程序段頭中的filesz可能比memsz小，這表示中間相差的地方應該用0填充（對於C的全局變量）而不是繼續從文件中讀數據。對於/init，filesz是2240字節而memsz是2252字節。所以allocuvm會分配足夠的內存來裝2252字節的內容，但只從文件/init中讀取2240字節的內容。

現在exec要分配以及初始化用戶棧了。它只為棧分配一頁內存。exec一次性把參數字符串拷貝到棧頂，然後把指向它們的指針保存在ustack中。它還會在main參數列表argv的最後放一個空指針。這樣，ustack中的前三項就是偽造的返回PC，argc和argv指針了。

exec會在棧的頁下方放一個無法訪問的頁，這樣當程序嘗試使用超過一個頁的棧時就會出錯。另外，這個無法訪問的頁也讓exec能夠處理那些過於龐大的參數；當參數過於龐大時，exec用於將參數拷貝到棧上的函數copyout會發現目標頁無法訪問，並且返回-1。

在創建新的內存映像時，如果exec發現了錯誤，比如一個無效的程序段，它就會跳轉到標記bad處，釋放這段內存映像，然後返回-1。exec必須在確認該調用可以成功後才能釋放原來的內存映像，否則，若原來的內存映像被釋放，exec甚至都無法向它返回-1了。exec中的錯誤只可能發生在新建內存映像時。一旦新的內存映像建立完成，exec就能裝載新映像（5989）而把舊映像釋放（5990）。最後，exec成功地返回0。
